# Deploy to Ploi Cloud

Deploy the current project to Ploi Cloud directly from your editor. Scans the project, creates or updates the application via MCP, deploys, monitors, and auto-fixes failures.

## Prerequisites

The Ploi Cloud MCP server must be connected. If not configured, tell the user to run:

```
claude mcp add ploi-cloud --url https://ploi.cloud/mcp
```

## Phase 1: Git validation

These checks are mandatory. Abort with a clear message if any fail.

1. Run `git rev-parse --git-dir` ‚Äî abort if not a git repo
2. Run `git remote get-url origin` ‚Äî abort if no remote URL
3. Parse the remote URL to extract owner and repo name (supports github.com, gitlab.com, bitbucket.org)
4. Run `git branch --show-current` to get the current branch

## Phase 2: Project detection

Scan local files to determine the project type and requirements.

**Type detection (first match wins):**
- `composer.json` containing `laravel/framework` ‚Üí `laravel`
- `composer.json` containing `statamic/cms` ‚Üí `statamic`
- `composer.json` containing `craftcms/cms` ‚Üí `craftcms`
- `wp-config.php` exists ‚Üí `wordpress`
- `composer.json` exists (generic) ‚Üí `php`
- `package.json` exists ‚Üí `nodejs`

**Runtime detection:**
- PHP version: read `composer.json` ‚Üí `require.php` constraint, pick the highest matching version (8.1, 8.2, 8.3, 8.4)
- Node.js version: read `package.json` ‚Üí `engines.node`, default to `20` if not specified
- PHP extensions: collect all `ext-*` keys from `composer.json` ‚Üí `require`, exclude built-in (opcache, pcntl, pdo_mysql, pdo_pgsql, redis, sodium, zip)

**Service detection from dependencies:**

| Dependency | Service type |
|---|---|
| `mysql2`, `@prisma/client` with MySQL in schema | `mysql` |
| `pg`, `postgres`, `@prisma/client` with PostgreSQL | `postgresql` |
| `mongodb`, `mongoose`, `mongodb/mongodb` | `mongodb` |
| `predis/predis`, `ext-redis`, `redis`, `ioredis` | `redis` |
| `php-amqplib/php-amqplib`, `amqplib` | `rabbitmq` |

**Build commands by type:**

| Type | Default build command |
|---|---|
| `laravel` | `composer install --no-interaction --prefer-dist --optimize-autoloader --no-dev && npm ci && npm run build` |
| `statamic` | `composer install --no-interaction --prefer-dist --optimize-autoloader --no-dev && npm ci && npm run build` |
| `craftcms` | `composer install --no-interaction --prefer-dist --optimize-autoloader --no-dev` |
| `wordpress` | (none) |
| `php` | `composer install --no-interaction --prefer-dist --optimize-autoloader --no-dev` |
| `nodejs` | `npm ci && npm run build` |

**Init commands:**
- Laravel/Statamic: `php artisan migrate --force`
- Craft CMS: `php craft install/check && php craft migrate/all --no-interaction`
- Others: none

**Secrets:**
- Read `.env.example` if it exists
- Extract variable names that have empty or placeholder values
- Exclude auto-injected variables (see references/service-env-vars.md)
- Exclude `APP_KEY` (auto-generated by Ploi Cloud for Laravel apps)

## Phase 3: Check existing app

1. Call `applications_index` to list all apps
2. Search for an app whose repository URL matches the current project's remote URL
3. If found ‚Üí go to Phase 4b (update)
4. If not found ‚Üí go to Phase 4a (create)

## Phase 4a: Create application

1. `applications_store` ‚Äî create with name (repo name), type, repository URL, owner, name, branch
2. `applications_build-config_update` ‚Äî set build commands and init commands
3. `applications_settings_update` ‚Äî set health_check_path (/ for most types, /up for Statamic)
4. `applications_php-config_update` ‚Äî set PHP version and extensions (PHP apps only)
5. `v1_applications_services_store` ‚Äî create each detected service (memory: 512Mi, volume_size: 5Gi)
6. `applications_secrets_store` ‚Äî add required secrets only
7. Go to Phase 5

## Phase 4b: Update existing application

1. `applications_repository_update` ‚Äî update repo URL and branch if changed
2. `applications_build-config_update` ‚Äî update build commands if needed
3. `v1_applications_services_index` ‚Äî check existing services, add any missing via `v1_applications_services_store`
4. Go to Phase 5

## Phase 5: Deploy

1. Call `applications_deploy` to trigger a deployment
2. Note the deployment ID from the response
3. Go to Phase 6

## Phase 6: Monitor

Poll every 15 seconds for up to 10 minutes.

1. Call `applications_deployments_logs` with `tail: 100` to check build progress
2. Call `applications_status` to check if the app is live
3. If deployment status is `failed` or `cancelled` ‚Üí go to Phase 7 (Auto-fix)
4. If application status is `active` ‚Üí go to Phase 8 (Success)
5. If still building (deployment status `running` or app status `deploying`) ‚Üí wait 15 seconds and repeat

## Phase 7: Auto-fix

Maximum 5 retry attempts. Diagnose the failure and apply a fix.

**Diagnosis:**
1. Read deployment logs (`applications_deployments_logs` with `tail: 2000`)
2. Read application logs (`applications_logs`)
3. Check debug summary (`applications_debug_summary`)

**Fix patterns:**

| Error | Fix |
|---|---|
| OOMKilled / exit code 137 | `applications_resources_update` ‚Äî increase memory (1Gi ‚Üí 2Gi ‚Üí 4Gi) |
| Missing PHP extension | `applications_php-config_update` ‚Äî add the extension |
| Build failure (composer/npm) | `applications_build-config_update` ‚Äî adjust commands |
| Health check failure | `applications_settings_update` ‚Äî change health_check_path |
| Missing environment variable | `applications_secrets_store` ‚Äî add the variable |
| Database connection error | Check service exists, verify env var injection |

After applying a fix:
1. Call `applications_deploy` to redeploy
2. Return to Phase 6 (Monitor)

## Phase 8: Success

1. Call `applications_show` to get the auto-assigned domain
2. Confirm with `applications_debug_summary` that pods are healthy
3. Report to the user:
   - Application name
   - Live URL (the auto-assigned domain)
   - Application ID
   - Services created
   - Any fixes that were applied

## Critical rules

- NEVER add domains ‚Äî the platform auto-assigns a preview domain
- NEVER add `APP_KEY` to secrets ‚Äî Ploi Cloud generates it for Laravel apps
- NEVER hardcode database credentials in secrets ‚Äî services auto-inject them via ConfigMap
- Default memory is `1Gi`. Use `applications_resources_update` with the `memory_request` parameter to change it
- Service volume_size must be `"5Gi"` (Kubernetes quantity format with unit suffix)
- Each build command entry becomes a separate `RUN` in the Dockerfile ‚Äî chain dependent commands with `&&`
- `${VAR}` interpolation in secrets only resolves other user secrets, NOT service-injected vars
- Build commands run as root, but the application runtime runs as www-data (non-root). Build-time operations like `npm install -g` work fine
- Application status values: `starting`, `active`, `deploying`, `stopping`, `stopped`, `destroying`, `failed`. Use `active` to confirm the app is live
- Deployment status values: `pending`, `running`, `success`, `failed`, `cancelled`. Note that `running` means the build is in progress, not that the app is live
